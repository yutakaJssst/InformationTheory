## 8.1 なぜ大学院生がGitHubを使うべきか

### 8.1.1 研究コードのバージョン管理

大学院での研究では、コードを何度も修正する。バージョン管理なしでは、以下のような問題が生じる。

```
よくある悲劇:
analysis.py
analysis_v2.py
analysis_v2_final.py
analysis_v2_final_revised.py
analysis_v2_final_revised_REAL_FINAL.py
```

このようなファイル管理では、「どの時点で何を変更したか」「なぜ変更したか」が全く分からなくなる。Gitを使えば、一つのファイルの変更履歴を完全に追跡でき、いつでも過去の状態に戻れる。

### 8.1.2 共同研究での活用

研究は一人で行うものではない。指導教員との共同作業、研究室のメンバーとのコード共有、他大学の共同研究者とのプロジェクトなど、複数人での作業は日常的に発生する。

GitHubを使えば、以下が可能になる。

- **コードの共有**: URLを送るだけで共同研究者がコードにアクセスできる
- **同時作業**: 複数人が同じプロジェクトを同時に編集できる
- **変更の追跡**: 誰が、いつ、何を変更したかが一目で分かる
- **レビュー**: 共同研究者の変更をレビューしてからマージできる

### 8.1.3 就職活動でのポートフォリオ効果

技術系の就職活動において、GitHubのプロフィールは「第二の履歴書」として機能する。特にIT企業やデータサイエンス関連のポジションでは、GitHubでの活動が評価される。

**GitHubプロフィールで評価されるポイント:**

- 定期的なコミット（継続的に開発・研究している証拠）
- README.md が整備されたリポジトリ（ドキュメンテーション能力）
- コードの品質（命名規則、コメント、テストの有無）
- オープンソースプロジェクトへの貢献

### 8.1.4 オープンサイエンスへの貢献

近年、研究のオープン化（Open Science）が世界的なトレンドとなっている。論文のデータや分析コードをGitHubで公開することで、以下のメリットがある。

- **研究の透明性向上**: 分析手法が第三者に検証可能になる
- **再現性の担保**: 他の研究者が同じ分析を再現できる
- **引用の増加**: 公開されたコードやデータは他の研究者に利用され、引用が増える傾向がある
- **学術コミュニティへの貢献**: 研究ツールやデータセットの共有

## 8.2 Gitの基礎

### 8.2.1 バージョン管理とは何か

バージョン管理とは、ファイルの変更履歴を記録し、過去の任意の時点の状態を復元できるようにする仕組みである。

**身近な例:**

- Google ドキュメントの「変更履歴」機能
- Microsoft Word の「変更の追跡」機能
- ゲームの「セーブポイント」

Gitは、これらの機能をより強力に、より体系的に提供するツールである。

### 8.2.2 リポジトリ、コミット、ブランチの基本概念

Gitを理解するための3つの核心概念を、ゲームのアナロジーで説明する。

**リポジトリ（Repository）= セーブデータフォルダ**

リポジトリとは、プロジェクトの全ファイルとその変更履歴を格納する場所である。ゲームで言えば、セーブデータが保存されるフォルダに相当する。

```
my-research-project/     ← これがリポジトリ
├── .git/                ← Git の管理情報（変更履歴が格納される）
├── analysis.py          ← 研究のコード
├── data/                ← データフォルダ
│   └── experiment.csv
├── figures/             ← グラフフォルダ
│   └── result.png
└── README.md            ← プロジェクトの説明
```

**コミット（Commit）= セーブポイント**

コミットとは、ある時点でのファイルの状態を保存する操作である。ゲームでセーブポイントを作るのと同じだ。

```
セーブポイントの例:
[コミット1] "初期セットアップ: データ読み込みコードを作成"
     ↓
[コミット2] "前処理を追加: 欠損値の補完"
     ↓
[コミット3] "統計分析を実装: t検定による群間比較"
     ↓
[コミット4] "グラフ作成: 結果の棒グラフを追加"
     ↓
[コミット5] "論文用に図表を修正"
```

各コミットには「何を変更したか」を説明するメッセージを付ける。後から見返したときに、変更の意図が分かるようにすることが重要である。

**ブランチ（Branch）= 並行世界**

ブランチとは、メインの開発ラインから分岐して、独立に作業できる仕組みである。ゲームで言えば、「異なるルートを試す」ことに相当する。

```
main ─────●─────●─────●─────●──── (メインの開発ライン)
                 │                ↑
                 └──●──●──●──────┘  feature/new-analysis
                   (新しい分析手法を試す)
```

新しい分析手法を試したいとき、ブランチを作れば、メインのコードを壊す心配なく実験できる。うまくいったらメインに統合（マージ）し、うまくいかなければ捨てれば良い。

### 8.2.3 「セーブポイント」のアナロジーによる理解

| Git 用語 | ゲームのアナロジー | 説明 |
|---------|-----------------|------|
| リポジトリ | セーブデータフォルダ | プロジェクト全体の格納場所 |
| コミット | セーブポイント | ある時点の状態を記録 |
| ブランチ | 並行ルート | 分岐して独立に作業 |
| マージ | ルートの合流 | 分岐した作業を統合 |
| プッシュ | クラウドに同期 | ローカルの変更をGitHubに送る |
| プル | クラウドから同期 | GitHubの変更をローカルに取得 |
| クローン | セーブデータをコピー | GitHubからプロジェクトをダウンロード |

## 8.3 GitHubの基本操作

### 8.3.1 GitHubアカウントの作成

1. https://github.com にアクセスする
2. 「Sign up」をクリックし、メールアドレス・ユーザー名・パスワードを入力する
3. メール認証を完了する

**ユーザー名の選び方:**
- 本名またはそれに近い名前が望ましい（就職活動で使うため）
- 例: `yutaka-tanaka`, `tanaka-lab`, `ytanaka-research`
- 避けるべき: `cool_hacker_2024`, `xxx_anonymous` など

**GitHub Education への登録（推奨）:**
- https://education.github.com にアクセス
- 大学のメールアドレスで学生認証を行う
- GitHub Copilot が無料で使える
- プライベートリポジトリが無制限に作成可能

### 8.3.2 GitHub Desktop のセットアップ

コマンドラインが苦手な人には、GitHub Desktop（GUIアプリ）が推奨される。

**インストール手順:**

1. https://desktop.github.com からダウンロード
2. インストール後、GitHub アカウントでサインイン
3. デフォルトエディタを VS Code に設定

**GitHub Desktop の画面構成:**

```
┌─────────────────────────────────────────┐
│  Current Repository ▼ │ Current Branch ▼  │
├─────────────────────────────────────────┤
│  Changes          │  History             │
│  ┌──────────┐     │  ┌──────────┐        │
│  │ 変更された │     │  │ コミット  │        │
│  │ ファイル   │     │  │ 履歴一覧  │        │
│  │ の一覧    │     │  │          │        │
│  └──────────┘     │  └──────────┘        │
│                   │                      │
│  Summary          │                      │
│  [コミットメッセージ入力欄]              │
│                   │                      │
│  [Commit to main] │                      │
└─────────────────────────────────────────┘
```

### 8.3.3 リポジトリの作成とクローン

**GitHub上でリポジトリを作成する手順:**

1. GitHub にログイン
2. 右上の「+」→「New repository」をクリック
3. 以下を入力:
   - Repository name: `my-research-project`
   - Description: 「修士研究のデータ分析コード」
   - Public または Private を選択
   - 「Add a README file」にチェック
4. 「Create repository」をクリック

**GitHub Desktop でクローンする手順:**

1. GitHub Desktop を開く
2. 「File」→「Clone repository」を選択
3. GitHubのリポジトリ一覧から対象を選択
4. ローカルの保存先フォルダを指定
5. 「Clone」をクリック

### 8.3.4 コミット → プッシュの基本フロー

日常的な作業の流れは以下の通りである。

```
Step 1: ファイルを編集する（VS Code等で）
    ↓
Step 2: 変更をステージングする（保存したい変更を選択）
    ↓
Step 3: コミットメッセージを書く
    ↓
Step 4: コミットする（ローカルにセーブ）
    ↓
Step 5: プッシュする（GitHubにアップロード）
```

**GitHub Desktop での操作:**

```
1. ファイルを編集して保存すると、左側の「Changes」に変更が表示される
2. コミットに含めたいファイルにチェックを入れる
3. 「Summary」欄にコミットメッセージを入力
   例: 「データ前処理の欠損値処理を追加」
4. 「Commit to main」ボタンをクリック
5. 上部の「Push origin」ボタンをクリック
```

**良いコミットメッセージの書き方:**

| 悪い例 | 良い例 |
|-------|-------|
| 「更新」 | 「t検定の結果を棒グラフで可視化」 |
| 「修正」 | 「欠損値処理のバグを修正: NaNが残る問題」 |
| 「追加」 | 「相関分析のコードとヒートマップを追加」 |
| 「あああ」 | 「README.mdに実行手順を追記」 |

コミットメッセージは「何を、なぜ変更したか」が分かるように書く。将来の自分が読んで理解できることを意識する。

### 8.3.5 VS Code との連携

VS Code には Git の操作機能が組み込まれている。

**VS Code での Git 操作:**

1. 左サイドバーの「ソース管理」アイコン（分岐のようなアイコン）をクリック
2. 変更されたファイルが一覧表示される
3. ファイル名の横の「+」をクリックしてステージング
4. 上部のテキストボックスにコミットメッセージを入力
5. チェックマークボタンでコミット
6. 「...」メニューから「プッシュ」を選択

**VS Code の Git 拡張機能（推奨）:**

- **GitLens**: コードの各行が誰によっていつ変更されたかを表示
- **Git Graph**: コミット履歴をグラフで可視化

## 8.5 GitHub Pagesで公開する

### 8.5.1 GitHub Pagesとは

GitHub Pages は、GitHubのリポジトリに置いたHTMLファイルを無料でWebサイトとして公開できるサービスである。

**特徴:**
- 無料で利用可能
- `https://ユーザー名.github.io/リポジトリ名/` のURLで公開される
- リポジトリにプッシュするだけで自動的に更新される
- カスタムドメインも設定可能

### 8.5.2 GitHub Pages の設定手順

**手順1: リポジトリの準備**

1. GitHubで新しいリポジトリを作成する
   - リポジトリ名: `my-portfolio`（または `ユーザー名.github.io`）
   - Public に設定する（GitHub Pages は Public リポジトリで無料）
2. リポジトリに `index.html` をアップロードする

**手順2: GitHub Pages の有効化**

1. リポジトリのページで「Settings」タブをクリック
2. 左メニューの「Pages」をクリック
3. 「Source」で以下を設定:
   - Branch: `main`
   - Folder: `/ (root)`
4. 「Save」をクリック

**手順3: URLの確認**

設定後、数分待つと以下のURLでサイトが公開される。

```
https://ユーザー名.github.io/my-portfolio/
```

ページ上部に「Your site is live at ...」というメッセージとURLが表示される。

### 8.5.3 更新フロー

Webページを更新する場合、以下のフローで行う。

```
1. ローカルで index.html を編集する
    ↓
2. Live Server でプレビュー確認
    ↓
3. GitHub Desktop でコミット
   コミットメッセージ例: 「研究業績に国際会議の発表を追加」
    ↓
4. プッシュする
    ↓
5. 数分後にWebサイトに反映される
```

**Tips:**
- 反映には通常1〜3分程度かかる
- キャッシュが残っている場合は、ブラウザの強制リロード（Ctrl+Shift+R）を試す
- GitHub の Actions タブでデプロイの状況を確認できる

## 8.6 研究ポートフォリオの構築

### 8.6.1 ポートフォリオに含めるもの

研究者ポートフォリオには、以下の要素を含めることを推奨する。

| セクション | 内容 | 重要度 |
|-----------|------|--------|
| 自己紹介 | 名前、所属、研究分野、顔写真 | 必須 |
| 研究概要 | 研究テーマの説明（一般向け） | 必須 |
| 業績一覧 | 論文、学会発表、受賞歴 | 必須 |
| スキル | プログラミング言語、ツール、資格 | 推奨 |
| プロジェクト | 研究プロジェクトの詳細と成果物 | 推奨 |
| 連絡先 | メールアドレス、SNS | 必須 |
| CV/履歴書 | PDF形式でダウンロード可能 | 推奨 |

### 8.6.2 README.md の書き方

README.md は、リポジトリの「顔」である。GitHubでリポジトリを開いたとき最初に表示されるため、丁寧に書くことが重要である。

**良い README.md の構成:**

```markdown
# プロジェクト名

## 概要
このプロジェクトが何をするものかを1-2文で説明。

## 特徴
- 特徴1
- 特徴2
- 特徴3

## インストール方法
```bash
pip install -r requirements.txt
```

## 使い方
```python
from my_module import analyze
result = analyze("data.csv")
```

## ファイル構成
```
project/
├── src/           # ソースコード
├── data/          # データ（サンプルのみ）
├── figures/       # 生成されたグラフ
├── tests/         # テストコード
├── requirements.txt
└── README.md

## 8.7 共同研究でのGitHub活用

### 8.7.1 Issue による課題管理

Issue は、プロジェクトの課題やタスクを管理する機能である。研究プロジェクトでは、以下のような使い方ができる。

**Issue の使用例:**

```
Issue #1: データの前処理パイプラインを作成する
  - [ ] CSVファイルの読み込み
  - [ ] 欠損値の処理
  - [ ] 外れ値の検出と対応
  - [ ] 前処理済みデータの保存
  担当: @tanaka
  期限: 2026-04-15

Issue #2: 統計分析の実装
  - [ ] 記述統計量の算出
  - [ ] 群間比較（t検定）
  - [ ] 相関分析
  担当: @yamada
  期限: 2026-04-22

Issue #3: 図表の作成
  - [ ] Figure 1: 群間比較の棒グラフ
  - [ ] Figure 2: 相関の散布図
  - [ ] Table 1: 記述統計量
  担当: @tanaka
  期限: 2026-04-30
```

**Issue の作成手順（GitHub上）:**

1. リポジトリの「Issues」タブをクリック
2. 「New issue」をクリック
3. タイトルと説明を記入
4. ラベル（bug, enhancement, question等）を付与
5. 担当者（Assignees）を指定
6. 「Submit new issue」をクリック

### 8.7.2 Pull Request によるコードレビュー

Pull Request（PR）は、ブランチの変更をメインブランチに統合する前に、レビューを行う仕組みである。

**Pull Request のワークフロー:**

```
1. 新しいブランチを作成する
   例: feature/add-correlation-analysis

2. ブランチ上でコードを書いてコミットする

3. GitHub上で Pull Request を作成する
   - タイトル: 「相関分析機能の追加」
   - 説明: 変更内容、テスト結果、関連するIssue番号

4. 共同研究者がコードをレビューする
   - コメントで改善提案
   - 「Approve」で承認

5. レビューが通ったらマージする
```

**GitHub Desktop での新しいブランチの作成:**

1. 上部の「Current Branch」をクリック
2. 「New Branch」をクリック
3. ブランチ名を入力（例: `feature/add-correlation-analysis`）
4. 「Create Branch」をクリック
5. このブランチ上でコードを編集・コミットする
6. 「Publish branch」でGitHubにプッシュ
7. GitHub上で「Compare & pull request」をクリック

### 8.7.3 研究コードの共有とライセンス

研究コードを公開する場合、適切なライセンスを選択することが重要である。

**主なオープンソースライセンス:**

| ライセンス | 特徴 | 推奨場面 |
|-----------|------|---------|
| MIT | 最も自由。商用利用・改変・再配布可能 | 研究コードの一般公開 |
| Apache 2.0 | MIT + 特許条項。企業でも使いやすい | ツール・ライブラリの公開 |
| GPL v3 | 派生物も同じライセンスで公開必須 | 強いオープンソースを望む場合 |
| CC BY 4.0 | クリエイティブ・コモンズ。主にデータ向け | データセットの公開 |

**推奨**: 研究コードには **MIT ライセンス** が最も広く使われている。迷ったらMITを選べば間違いない。

**ライセンスの追加方法:**

1. GitHubのリポジトリページで「Add file」→「Create new file」
2. ファイル名に `LICENSE` と入力
3. 右側に「Choose a license template」ボタンが表示される
4. テンプレートを選択して作成
