## 第4回課題（文献調査）の総評

### 提出状況

- 現時点で <span class="teal">**24ファイル**</span> を確認
- 同一内容の重複らしきファイル、プロンプトログの別添を含むため、人数ベースでは暫定
- ほとんどの提出で、研究テーマ・関連論文・新規性まで到達していた

### 全体として良かった点

- AIを単なる要約係ではなく、<span class="highlight">キーワード拡張・論文候補探索・新規性整理</span>に使えていた
- 自分の研究テーマにかなり具体的なContextを与えている提出が多かった
- Perplexity、Gemini、ChatGPT、Claude、Google Scholar、arXivなどを組み合わせた提出が見られた

---

## 参考になった工夫（1/2）

### 工夫① <span class="teal">AIツールの役割を分ける</span>

検索はPerplexityやGoogle Scholar、要約・比較はGeminiやChatGPT、長い本文の整理はClaudeのように、<span class="highlight">用途ごとにAIを使い分ける</span>提出があった。これは文献調査として実践的。

### 工夫② <span class="teal">「存在しない論文を作らない」と明示する</span>

プロンプトに「存在しない論文名・著者名・DOIは作らない」「URLやarXiv IDを示す」と入れている提出があった。AIによる文献調査では、この指定が重要。

### 工夫③ <span class="teal">英語キーワードまで広げる</span>

日本語の研究テーマをそのまま検索せず、英語キーワード・周辺分野・未解決課題まで広げていた提出は、文献候補の幅が広かった。

---

## 参考になった工夫（2/2）

### 工夫④ <span class="teal">各論文を「自分の研究との関連」まで戻す</span>

単なる論文要約で終わらず、「この論文は自分の研究のどの部分に効くのか」を書いている提出が良かった。文献調査は、論文を集める作業ではなく、<span class="highlight">自分の研究の位置づけを作る作業</span>。

### 工夫⑤ <span class="teal">新規性を比較軸で説明する</span>

「対象データが違う」「評価指標が違う」「実装環境が違う」「既存研究は検出精度、本研究は回避耐性を見る」のように、差分を軸で説明している提出は説得力があった。

### 工夫⑥ <span class="teal">AIとの対話ログを残す</span>

プロンプトと出力を残している提出は、どのように調査を進めたかが追跡できた。研究ノートとしても価値がある。

---

## 共通の落とし穴

### 落とし穴① <span class="red">AI出力の貼り付けが多すぎる</span>

プロンプトと出力を残すのは良いが、レポート本文では自分で整理する必要がある。AIの出力全文は、本文ではなく付録・別添に回すと読みやすい。

### 落とし穴② <span class="red">出典情報が不足する</span>

タイトルだけでは、後から論文を確認できない。最低限、<span class="highlight">著者・年・出版先・URL/DOI/arXiv ID</span> を残すこと。

### 落とし穴③ <span class="red">新規性が「まだやられていない」だけになる</span>

「やられていない」は証明が難しい。次のように書く方が強い。

> 既存研究はAを対象にしている。本研究はBを対象にし、Cという評価指標で比較する。

## なぜ英語論文が必要か

### 国際発信の重要性

- 主要な国際会議・ジャーナルの<span class="highlight">公用語は英語</span>
- 日本語で書いた論文は、世界の研究者に読まれにくい
- 被引用数、共同研究、キャリア形成に直結する

### AI時代に重要なこと

- AIは英訳を支援できるが、<span class="red">意味の正しさ</span>は保証しない
- 「この英語は何を意味しているか」を判断する力が必要
- 今日の演習は、AI出力を見抜くための基礎訓練

---

## AIにできること／できないこと

| AIにできること | AIだけでは危ないこと |
|---|---|
| 日本語から英語への翻訳 | 研究内容の正確性の保証 |
| 文法・スペルチェック | 専門用語の分野内での妥当性判断 |
| 表現の言い換え | 実験結果・考察の意味づけ |
| Abstract構造の提案 | 新規性・貢献の最終判断 |

> AIは「英訳者」ではなく、<span class="highlight">推敲相手</span>として使う。最終責任は著者にある。

# 技術英語の4原則

英訳する前に日本語を整える

---

## 技術英語の4原則

| 原則 | ねらい | AIに渡す前の確認 |
|---|---|---|
| **すっきり最小主義** | 冗長な表現を削る | 空っぽ語はないか |
| **具体化主義** | 曖昧さを数値・条件に変える | 「高い」「多い」は具体的か |
| **動詞主義** | 名詞句を動詞に戻す | 「〜を行った」で逃げていないか |
| **能動態主義** | 主語と責任を明確にする | 誰が何をしたか言えるか |

> 良い英訳は、良い日本語からしか生まれない

# 論文英語で間違いやすい文法

<!-- _class: ai-exercise compact -->

## <span class="ai-badge">問題5</span> 単純現在と進行形

次の英文を訳し分けてください。

<div class="question-box">
1. I clean the room.<br>
2. I am cleaning the room.<br>
3. This method reduces noise.<br>
4. This method is reducing noise.
</div>

### 考えるポイント

- 単純現在は「いつもそう」「再現性がある」
- 進行形は「いま起きている」「そのうち終わる」

<span class="timebox">3分</span>


---

<!-- _class: ai-exercise compact -->

## <span class="ai-badge">問題6</span> 現在完了と過去形

次の英文を訳し分けてください。

<div class="question-box">
1. This machine has failed three times.<br>
2. This machine failed three times yesterday.<br>
3. We have evaluated the method using 120 images.<br>
4. We evaluated the method using 120 images.
</div>

### 考えるポイント

- 現在完了は「現在につながる経験・完了・状態」
- 過去形は「過去のある時点で起きた事実」

<span class="timebox">3分</span>


---

<!-- _class: ai-exercise compact -->

## <span class="ai-badge">問題7</span> when / while / 現在完了

次の英文を訳し分けてください。

<div class="question-box">
1. Push the button when the light turns on.<br>
2. Push the button while the light is blinking.<br>
3. Push the button when the light has turned on.<br>
4. The paper turns red when the solution is acidic.
</div>

### ヒント

`solution`: 溶液 ／ `paper`: 試験紙 ／ `blink`: 点滅する

<span class="timebox">3分</span>


---

<!-- _class: ai-exercise compact -->

## <span class="ai-badge">問題8</span> 助動詞でニュアンスを足す

次の英文のニュアンスを説明し、日本語にしてください。

<div class="question-box">
1. If you tap No, the processing will abort.<br>
2. If the system is overloaded, the process may abort.<br>
3. If necessary, the process can be aborted manually.<br>
4. The results suggest that the proposed method improves accuracy.
</div>

### 考えるポイント

- `will`: 条件を満たせばそうなる、という強い見通し
- `may`: 可能性があるが、起きるとは限らない
- `can`: 可能である、または機能としてできる

<span class="timebox">3分</span>


---

<!-- _class: ai-exercise compact -->

## <span class="ai-badge">問題9</span> 冠詞 a / the

空欄に `a`, `the`, または空欄のままを入れてください。

<div class="question-box">
1. We propose ( ) method for detecting unattended baggage.<br>
2. ( ) method consists of three modules.<br>
3. Open ( ) new window. Drag images onto ( ) new window.<br>
4. Select ( ) files in the SOURCE window.
</div>

### 考えるポイント

- 初出か、2回目以降か
- 後ろから限定されているか
- 「どれでもよい」のか「その全部」なのか

<span class="timebox">3分</span>


---

<!-- _class: ai-exercise compact -->

## <span class="ai-badge">問題10</span> 他動詞と自動詞

次の英文の意味を説明し、必要なら直してください。

<div class="question-box">
1. I will submit next Wednesday.<br>
2. I will submit it next Wednesday.<br>
3. We are searching the lost luggage, but we cannot find it.<br>
4. We are searching for the lost luggage, but we cannot find it.
</div>

### 考えるポイント

- `submit` は目的語が必要か
- `search` と `search for` は同じか
- 辞書で「自動詞」「他動詞」を見る

<span class="timebox">3分</span>

# AIを使ったAbstract作成

英語の判断軸を持ったうえでAIを使う

---

## Abstractの基本構造

### 150-200語で、論文全体を圧縮する

| 要素 | 文数 | 書く内容 |
|---|---:|---|
| 背景 | 1文 | 分野の課題・重要性 |
| 目的 | 1文 | 本研究が何を目指すか |
| 手法 | 1-2文 | どのようなアプローチか |
| 結果 | 1-2文 | 主な結果、できれば数値 |
| 意義 | 1文 | 貢献・応用可能性 |

> Abstractは「短いIntroduction」ではなく、論文全体の要約

---

## AI英訳ワークフロー

### 4つのStep

1. **日本語を構造化する**  
   背景・目的・手法・結果・意義に分ける
2. **セクション単位で英訳する**  
   いきなり全体を丸投げしない
3. **技術英語の4原則で推敲する**  
   冗長さ、曖昧さ、名詞句、受動態を確認
4. **文法・用語を点検する**  
   時制、冠詞、助動詞、専門用語を確認

> 先ほどの演習は、Step 3-4の判断軸になる

---

## Abstractチェックリスト

提出前に、次を確認する。

- 背景・目的・手法・結果・意義が入っている
- 結果に数値または評価条件が入っている
- AIが追加した事実が混ざっていない
- `prove`, `perfectly`, `always` などの強すぎる表現がない
- 手法・結果の時制が一貫している
- 初出の名詞に `a/an`、2回目以降に `the` が適切に使われている
- 専門用語が自分の分野の論文で実際に使われている
